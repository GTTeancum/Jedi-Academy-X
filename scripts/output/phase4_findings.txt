5. MEMORY (Hunk_*/Z_*)
════════════════════════════════════════════════════════════════════════════════

Z_Init @ 0x49945 (107 insns, complete):
  1. Clears zone header at .bss 0x7BC2F0 (0x100 bytes / 64 DWORDs)
  2. Clears zone bitmap at .bss 0x7AC2F0 (0x10000 bytes / 16384 DWORDs)
  3. Queries physical RAM via fn @ 0xC2582:
     - Calls ord_181 (XGetPhysicalMemoryStatus / MmQueryStatistics via [0x2CD0EC])
     - Extracts used_pages × 4096 and free_pages × 4096
  4. Prints "*** PhysRAM: %d used, %d free"
  5. Calls D3D @ 0x23F620 = D3D_AllocContiguousMemory(0x1000000, 0)
     - Allocates exactly 16MB contiguous physical memory
     - Result stored at [0x81CE18] (zone pool base)
     - Zone pool size 0x1000000 stored at [0x81CE20]
     - [0x81CE1C/24/28/2C] = 0 (Hunk low/high marks)
  6. Queries physical RAM again (prints second PhysRAM line)
  7. Computes zone end: [esp+0x18] + 0xFF930000 (= base + 16MB rounding)
  8. Calls 0xC268B to allocate zone header struct from pool
     - Zone header fields: self-ptr, size, free-list sentinel ptrs
     - Free-list initialized at .bss 0x7AC1C0+
  9. Registers cvars/cmds: zone_stats, zone_details, zone_memmap, zone_cstats
  10. Calls 0xC1A23 = NtCreateSemaphore wrapper for zone mutex
  11. Calls 0xEC6D0 = sound pool allocator:
      - Allocates 0x139000 (1.23MB) via Z_Malloc-equivalent @ 0x49C90
      - Stores sound pool ptr at [0xA05480]

Z_Malloc @ 0x49B00:
  - Standard VV zone allocator (linked-list based, first-fit)
  - Overflow buffer for small free blocks
  - Zone counters: [0x7BC300]=free count, [0x7BC304]=free size

MmAllocateContiguousMemory: 0 callers (D3D uses D3D_AllocContiguousMemory internally)
MmAllocateContiguousMemoryEx: 1 caller at 0xC1E29 (XAPI internal, 0x1000 bytes)
ExAllocatePool: many callers (XAPI runtime / C runtime, not game code)
0x1000000 constant: at 0x499AC only (Z_Init)

MATCH vs SOURCE (code/qcommon/z_memman_console.cpp):
  Shipped binary: D3D_AllocContiguousMemory(16MB) for zone pool ← key difference
  Our source:     gTextures.Initialize(16/20MB) → D3D_AllocContiguousMemory for TEXTURES
                  VirtualAlloc(NULL, dynamic_size) for zone pool

  Analysis: The shipped binary uses D3D contiguous memory for the zone pool.
  Our source uses VirtualAlloc for the zone pool and D3D_AllocContiguousMemory
  for the texture pool (via StaticTextureAllocator::Initialize).

  VirtualAlloc on Xbox maps to NtAllocateVirtualMemory (kernel thunk ordinal 183
  is in the thunk table). This should work correctly on retail hardware.
  The zone pool doesn't need to be physically contiguous — it's CPU-only.

  CONCLUSION: Difference is architecture-level (VV vs shipped layout) but
  functionally equivalent. VirtualAlloc will succeed on Xbox. No patch needed.

  CAVEAT: Our source also calls gTextures.Initialize INSIDE Z_Init which calls
  D3D_AllocContiguousMemory. This is safe: D3D_AllocContiguousMemory does NOT
  require an active D3D device (it's just MmAllocateContiguousMemory internally).
  Z_Init is called before GLW_Init/CreateDevice — this is acceptable.

ACTION REQUIRED: None — memory layout difference is benign.

HDTV detection at GLW_Init (0xA4249 → 0xC26A8):
  - 0xC26A8 tests a bit in HDTV mode register
  - If bit 8 set: flags += 0x10 (DPRESENT_PROGRESSIVE), HDTV path
  - Our source code handles HDTV via glw_state->isWidescreen check
  - Binary behavior is equivalent

Zone pool size discrepancy:
  - Shipped: fixed 16MB (D3D_AllocContiguousMemory)
  - Rebuilt: dynamic (VirtualAlloc, capped at 64MB)
  - Xbox retail has 64MB total; D3D reserves ~32MB for GPU textures/render targets
  - Available for game: ~32MB
  - Our 64MB cap will trigger → actual alloc ~32MB (more than shipped 16MB)
  - Could cause issues if physical memory is fragmented, but VirtualAlloc uses
    virtual address space which is less fragmented
